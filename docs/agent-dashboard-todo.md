# 에이전트(셀러) 대시보드 — 구현 TODO

> 여기서 "에이전트(agent)"는 `UserRole.AGENT`, 즉 플랫폼에 입점한 **판매자(셀러)** 를 말한다.
> AI 코딩 에이전트 지침 파일(`AGENTS.md` 관례)과는 무관하므로, 혼동을 피하기 위해 이 파일은
> `docs/agent-dashboard-todo.md`로 두고 기존 [`agent-dashboard-api.md`](./agent-dashboard-api.md)
> 옆에 배치했다.

## 이 문서의 목적

사용자가 제시한 9개 모듈(판매 실적/트래픽/상품 성과/고객 프로필/경쟁 동향/재고 물류/평판/광고 효과/AI 고급 기능)은
Amazon·Alibaba·Tmall급 셀러 센터를 기준으로 한 **이상적인 전체 목록**이다. 이 문서는 그 목록을
그대로 작업 목록화하지 않는다 — 대신 각 항목을 **이 프로젝트의 실제 백엔드가 가진 데이터로 지금 만들
수 있는지**를 기준으로 3단계로 나눈다.

- **Tier 1 (즉시 가능)** — 필요한 데이터가 이미 DB/이벤트에 존재. 집계·API·화면만 만들면 됨.
- **Tier 2 (선행 인프라 필요)** — 새로운 이벤트 수집 파이프라인이 있어야 함 (주로 클릭스트림: 노출/클릭/PV/UV).
  여러 모듈에 걸쳐 반복 등장하므로 **한 번만** 구축하는 공용 선행 과제로 취급한다.
- **Tier 3 (범위 밖 / 별도 서브시스템)** — 광고 캠페인 시스템, 인구통계 수집, 외부 경쟁사·키워드 피드,
  AI 예측/자동입찰, 다중 스토어·다국가 통합처럼 그 자체로 수 주 단위 별도 프로젝트인 것들.
  지금은 작업 항목이 아니라 "나중에 사업적으로 필요해지면 검토" 표시만 해둔다.

CLAUDE.md의 오프라인·온프레미스 제약(외부 광고 네트워크·PG·서드파티 API 연동 불가) 때문에,
Tier 3 상당수는 이 프로젝트 구조상 데이터 소스 자체가 없다는 점도 함께 표시한다.

## Phase 1 — 지금 바로 착수 (기존 실 데이터 → 모바일 배선)

가장 먼저 할 일이자 가장 가성비 높은 작업. **이미 완성되어 main에 있는 백엔드 5개 요약 API**
([`agent-dashboard-api.md`](./agent-dashboard-api.md) 참고)를 모바일 앱의 Agent 대시보드에 연결하는 것.
현재 `AgentHomeFragment`/`AgentEarningsFragment`는 정적 플레이스홀더 수치를 쓰고 있는데, 이걸 실 데이터로
교체하면 별도 백엔드 작업 없이 "진짜 작동하는" 셀러 대시보드가 완성된다.

- [ ] `GET /orders/agent/summary` → Agent Home의 매출/주문 통계 타일에 연결 (현재는 목록 개수만 셈)
- [ ] `GET /payments/settlements/summary` → Agent Earnings 화면의 정적 수익 카드/차트를 실 데이터로 교체
- [ ] `GET /inventory/agent/summary` → Agent Home에 "품절/임박 재고" 경고 타일 추가
- [ ] `GET /deliveries/my/summary` → Agent Home에 "지금 처리해야 할 일"(`actionNeeded`) 타일 추가
- [ ] `GET /products/catalog/variants/{variantId}/buybox` → 상품 수정 화면에 "Buy Box 순위/승리가" 표시
- [ ] 4개 요약 API는 병렬 호출이 전제이므로 Retrofit 병렬 호출 + 로딩/에러 상태 통합 처리

## Tier 2 선행 과제 — 클릭스트림 수집 파이프라인 (한 번만 구축)

모듈 2 전체, 모듈 3의 노출/클릭, 모듈 4의 행동 데이터, 모듈 8의 ROAS/CPC를 모두 이 하나의
인프라가 동시에 해결한다. 중복 설계하지 말 것.

- [ ] 이벤트 스키마 정의: `product_view`, `product_click`, `add_to_cart`(이미 존재하는 카트 이벤트 재사용),
      `search_query`(이미 Redis에 일부 존재) 등
- [ ] 수집 방식 결정: 프런트/모바일에서 Kafka 토픽으로 직접 발행 vs. 경량 수집 API(gateway 뒤에 신설)
- [ ] CQRS 원칙에 맞춰 sync-worker 또는 신규 `analytics-service`가 소비 → 집계 저장소(OpenSearch 또는 신규 Postgres 스키마)
- [ ] UV/PV, 유입 채널(referrer/UTM) 태깅 규칙
- [ ] 위 데이터가 쌓인 뒤에야 모듈 2/3/4/8의 Tier 1 승격이 가능
